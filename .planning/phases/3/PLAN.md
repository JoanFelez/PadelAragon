# Phase 3, Plan 1: Final Verification, Version Bump, and Commit

## Objective
Lock the finished suite into a releasable state, then bump the patch version and create a clean commit only after the full local verification path is green.

## Context
@.planning/ROADMAP.md
@.planning/STATE.md
@.planning/codebase/STACK.md
@.planning/codebase/CONCERNS.md
@app/build.gradle.kts

## Task Notes
- If you encounter an authentication/authorization error during execution (OAuth, API key, SSO, etc.), stop immediately and return a checkpoint requesting the user to authenticate.
- Do not stage unrelated pre-existing release artifacts unless they were intentionally regenerated by this task and reviewed.

## Tasks

### Task 1: Run final pre-bump verification
- **type:** auto
- **files:** app/build.gradle.kts, app/src/test/, app/src/main/java/
- **action:** Execute the agreed local verification path on the completed implementation before touching the app version.
- **verify:** `./gradlew clean :app:testDebugUnitTest :app:assembleDebug`
- **done:** The final implementation is green before any release/version metadata changes.

### Task 2: Bump patch version and rerun smoke verification
- **type:** auto
- **files:** app/build.gradle.kts
- **action:** Change the app version from `1.2.0` to `1.2.1` by incrementing `versionPatch`, keeping derived metadata consistent, then rerun a lightweight final verification on the version-bumped tree.
- **verify:** `./gradlew :app:testDebugUnitTest :app:assembleDebug`
- **done:** `app/build.gradle.kts` reports `1.2.1`, and the version bump is proven not to break the build or suite.

### Task 3: Review, stage, and commit only the intended change set
- **type:** checkpoint:human-verify
- **files:** app/build.gradle.kts, app/src/main/java/, app/src/test/, app/src/test/resources/, app/release/
- **action:** Review the diff for unrelated workspace changes, confirm the post-bump tree is still the green tree you intend to preserve, stage only the intended compile/test/version files, and create one final commit after that review. Use a clear message such as `test: add automated suite and bump version to 1.2.1`.
- **verify:** `git --no-pager status --short && git --no-pager diff --cached --stat && git commit -m "test: add automated suite and bump version to 1.2.1" && git --no-pager log -1 --pretty=%s`
- **done:** A single reviewed commit exists for the already-green 1.2.1 tree, the commit message clearly states suite + version-bump scope, and unrelated pre-existing modifications are not accidentally included.

## Verification
- Required local gate: `./gradlew clean :app:testDebugUnitTest :app:assembleDebug`
- Optional extra gate if a device/emulator becomes available during execution: `./gradlew :app:connectedDebugAndroidTest`
- Post-bump gate on the exact tree to be committed: `./gradlew :app:testDebugUnitTest :app:assembleDebug`
- Final commit gate: `git --no-pager status --short && git --no-pager diff --cached --stat && git --no-pager log -1 --pretty=%s`

## Success Criteria
1. The full local suite and debug build pass on the version-bumped tree.
2. The version bump happens after, not before, the code/test work is already green.
3. The final commit contains only intended task files.
4. The work lands as one clearly named suite-and-version-bump commit created from the post-bump green tree.
